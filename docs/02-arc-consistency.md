# アーク整合性（Arc Consistency）

## 核心的アイディア

各セルの「ドメイン（取りうる値の集合）」を管理し、**ある値がそのセルに割り当て可能な合法的配置が1つも存在しない場合、その値をドメインから除去する**（これを arc consistent な状態と呼ぶ）。

行・列の制約を交互に伝播させることで、ラインソルバ単体では確定できなかったセルを追加で確定できる。

> **アーク整合性の定義**：行（または列）がアーク整合であるとは、各セルについて、ドメインに残っている各値に対して、そのセルがその値を持つような合法的な完全配置が少なくとも1つ存在すること。

## アルゴリズムの特徴

- **DP の逆走査**：ラインソルバで構築した DP テーブルを逆方向にたどり、各セルが参加できる合法的配置を直接列挙する
- **双方向の制約伝播**：行の情報が列の更新に使われ、列の情報が行の更新に使われる相互作用が収束するまで続く
- **単調性**：ドメインは縮小のみ（一度除去された値は復元されない）
- **時間計算量**：1行あたり O(n × k)（DP フェーズと同等）
- ラインソルバの反復適用と本質的には等価だが、ドメイン管理の観点から整理した概念的フレームワーク

## メリット

- ラインソルバより多くのセルを確定でき、バックトラッキングが必要なケースを大幅に減らす
- 矛盾（空ドメイン）を早期に検出できるため、バックトラッキング中の不要な探索を削減できる
- DP の派生なので時間計算量を変えずに実現できる（O(n × k) per line）

## デメリット

- ラインソルバだけでは解けない問題でも、アーク整合性だけでは解けないパズルが依然として存在する
- 行と列の独立した整合性チェックであるため、行列をまたぐ複雑な依存関係は捉えられない
- 実装はラインソルバより複雑（ドメイン管理・変更検出の仕組みが必要）

## アルゴリズムの概要

### 前提

- 各セルのドメインを `{FILLED, EMPTY}` で初期化（既知セルは単一要素）
- 変更があった行・列を追跡するキュー（または変更フラグ）を用意する

### ステップ 1: 初期キューの構築

すべての行・列をキューに追加する。

### ステップ 2: 伝播ループ

```
while queue is not empty:
    line = queue.pop()
    new_domains = run_line_solver(line, current_domains)

    for each cell in line:
        removed = current_domains[cell] - new_domains[cell]
        if removed is not empty:
            current_domains[cell] = new_domains[cell]
            if current_domains[cell] is empty:
                → 矛盾検出：この経路は解なし
            else:
                → 垂直方向の行（または列）をキューに追加
```

### ステップ 3: ラインソルバ内でのドメイン更新

各行（列）についてラインソルバを実行する際、以下を判定する：

- セル `j` に `FILLED` が残るか：`FILLED` が含まれる合法的配置が1つ以上存在するか
- セル `j` に `EMPTY` が残るか：`EMPTY` が含まれる合法的配置が1つ以上存在するか

DP テーブルを逆走査し、合法的な遷移をたどることでこれを O(n × k) で実現できる。

### ステップ 4: 終了判定

- **解決**：全セルのドメインが単一値 → 解が求まった
- **不動点**：キューが空になり、まだ未確定セルが残る → バックトラッキングへ
- **矛盾**：いずれかのドメインが空 → この仮定は誤り（バックトラッキングで戻る）
